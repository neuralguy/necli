# SSH — удалённые серверы

Инструмент `ssh` работает ТОЛЬКО с предварительно настроенными хостами (их добавляет пользователь через `/ssh`). Каждый хост имеет короткий **alias** — именно его передавай в аргумент `host`, НЕ `user@host` и НЕ IP. Список доступных alias'ов с их адресами приведён в конце этого скилла (он подставляется автоматически при загрузке).

Никогда не спрашивай у пользователя `user@host` или адрес сервера — если хост настроен, бери его alias из списка ниже и используй сразу.

## Что умеет инструмент

Аргумент `host` (alias) обязателен во всех режимах. Режимы:

1. Выполнить команду:
   `{"host": "<alias>", "command": "ls -la /app"}`

2. Загрузить файл на сервер (scp):
   `{"host": "<alias>", "upload": "local/path.yaml", "dest": "/remote/path.yaml"}`

3. Скачать файл с сервера:
   `{"host": "<alias>", "download": "/remote/path.log", "dest": "local/path.log"}`

Соединение переиспользуется (ControlMaster, ~300с) — несколько команд подряд к одному хосту дешёвы.

## Ограничения и защита

- Интерактивные команды заблокированы (`vim`, `top`, `nano`, вложенный `ssh` и т.п.) — они зависнут. Используй неинтерактивные аналоги (`cat`, `sed -i`, `systemctl --no-pager`).
- Хардблок: `rm -rf /`, `reboot`, `shutdown` — не выполнятся никогда.
- Опасные команды (`rm -rf <path>`, `systemctl restart/stop`, `DROP TABLE`, `truncate` и т.п.) при `confirm_dangerous=true` у хоста требуют подтверждения пользователя — будет показан запрос. Не пытайся обойти.
- НЕ редактируй файлы на сервере через интерактивный редактор. Правь локально и заливай через `upload`, либо неинтерактивным `sed`/`tee` в `command`.

## Типичный флоу деплоя

1. Поправь файл локально (write_file/patch_file).
2. Залей на сервер: `{"host": "<alias>", "upload": "src/configs/x.yaml", "dest": "/app/src/configs/x.yaml"}`.
3. Перезапусти сервис: `{"host": "<alias>", "command": "systemctl restart <service> --no-pager"}` (потребует подтверждения).
4. Проверь: `{"host": "<alias>", "command": "systemctl status <service> --no-pager -n 20"}`.

## Команды цепочкой

`&&` / `||` внутри одной `command` допустимы (это удалённый shell, ограничение «не чейнить» относится к локальному инструменту shell, не к ssh). Но предпочитай отдельные вызовы — соединение всё равно переиспользуется, а вывод читается по шагам.
