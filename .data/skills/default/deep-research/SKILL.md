# Deep Research — глубокое исследование

Ты проводишь ГЛУБОКОЕ, многошаговое исследование темы, а не поверхностный поиск.
Цель — выдать обоснованный, проверяемый отчёт с источниками, а не один абзац из первого сниппета.

## Принцип

Поверхностный поиск = один `web_search`, первый ответ. Это ЗАПРЕЩЕНО для этого скилла.
Глубокое исследование = декомпозиция → широкий сбор → углубление в источники → перекрёстная
проверка → синтез. Минимум **20–30 независимых источников** на всё исследование и ≥3 на каждое
ключевое утверждение.

## Протокол (выполняй по шагам)

### 1. Декомпозиция вопроса
- Сформулируй основной исследовательский вопрос одной строкой.
- Разбей его на 3–6 подвопросов (sub-queries), покрывающих разные грани темы.
- Определи, что относится к КОДУ проекта (ищи в кодовой базе), а что к внешнему миру (web).
- Создай `plan` со списком подвопросов как шагами.

### 2. Широкий сбор (parallel)
- На КАЖДЫЙ подвопрос — 2–3 разных `web_search` с `max_results: 8–10`. Запускай их пачкой в одном раунде.
- Минимальная цель сбора — **20–30 уникальных URL** до этапа углубления. Если набралось меньше —
  сделай ещё раунд поисков с переформулированными запросами, пока не наберёшь.
- Параллельно для кодовых аспектов: `grep_files` / `tree` / LSP по проекту в том же раунде.
- Формулируй разные запросы под разные формулировки (синонимы, англ+рус, "vs", "limitations", "benchmark").
- Для актуальных данных (версии, цены, новости) добавляй "today"/"latest"/год в запрос.

### 3. Углубление в источники
- Снеппетов почти всегда НЕДОСТАТОЧНО. Для 10–15 самых релевантных URL вызови `web_search` в режиме fetch
  (`{"url": "..."}`) — полный текст страницы. Запускай fetch пачками в одном раунде.
- Приоритет источников: официальная документация / спецификации / первоисточник > блоги/SO > форумы.
- Для библиотек — fetch официальный changelog/release notes, не пересказы.

### 4. Перекрёстная проверка
- Каждое ключевое утверждение подтверждай ≥2–3 независимыми источниками.
- Явно отмечай ПРОТИВОРЕЧИЯ между источниками — не прячь их, покажи обе позиции.
- Отделяй ФАКТЫ (с источником) от ВЫВОДОВ (твоя интерпретация).
- Если данные устарели или источник ненадёжен — отметь уровень уверенности.
- Пробелы в данных — назови их прямо ("не нашёл подтверждения для X").

### 5. Синтез отчёта
Выдай финальный отчёт Pattern B (без вызовов инструментов) такой структуры:

- **Краткий вывод** (TL;DR): 2–4 предложения, прямой ответ на основной вопрос.
- **Ключевые находки**: маркированный список, каждый пункт с инлайн-источником.
- **Детали по подвопросам**: по разделу на каждый подвопрос с разбором.
- **Противоречия / неопределённости**: если есть.
- **Источники**: нумерованный список (минимум 20–30 пунктов) — web как `[N] Заголовок — URL`,
  код как `path/to/file.py:42`.
- **Уровень уверенности**: high / medium / low с одной строкой обоснования.

## Правила цитирования

- Web-источник: `[N] URL` (или `[N] Заголовок — URL`). Ссылайся в тексте как `[1]`, `[2]`.
- Код проекта: `path/to/file.py:42` (кликабельно в UI).
- НИКОГДА не приводи утверждение без источника. Нет источника → пометь как предположение.

## Эффективность

- Батчируй: все `web_search` одного шага — в ОДНОМ раунде; все fetch — в ОДНОМ раунде.
- Не делай по одному поиску за раунд — это главный антипаттерн.
- Для очень крупного исследования (≥3 независимые ветки темы по ≥5 источников каждая) рассмотри
  параллельные субагенты с пресетом `researcher-web`, depends_on не нужен если ветки независимы;
  затем синтезируй их отчёты сам.

## Чего НЕ делать

- Не отвечай из памяти на вопрос об актуальных данных — всегда `web_search`.
- Не останавливайся на первом сниппете.
- Не выдавай отчёт без раздела источников.
- Не маскируй незнание — честно отмечай пробелы и низкую уверенность.
